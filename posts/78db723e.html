Docker-Compose

如果项目依赖更多的第三方软件，我们需要管理的容器就更加多，每个都要单独配置运行，指定网络。
这时候，我们使用 docker-compose 把项目的多个服务集合到一起，一键运行。

  • 如果你是安装的桌面版 Docker，不需要额外安装，已经包含了。
  • 如果是没图形界面的服务器版 Docker，你需要单独安装 安装文档

运行docker-compose检查是否安装成功

要把项目依赖的多个服务集合到一起，我们需要编写一个`docker-compose.yml`文件，描述依赖哪些服务

参考文档：https://docs.docker.com/compose/

详细解释

1. version: “3.7”

  • 含义：这一行指定了 Docker Compose 文件的版本号，这里是 “3.7”。
  • 作用：Docker Compose 的不同版本支持不同的语法和功能。使用 “3.7” 确保文件遵循 3.7 版本的规则，并与兼容的 Docker 引擎一起工作。

2. services:

  • 含义：services 是 Docker Compose 文件的核心部分，用于定义应用程序中的各个服务（每个服务通常对应一个容器）。
  • 作用：在这个文件中，定义了两个服务：app 和 redis，分别代表一个自定义应用和一个 Redis 实例。
a. app:（自定义应用服务）
  • build: ./
    • 含义：指定构建 Docker 镜像的上下文路径，这里是当前目录（./）。
    • 作用：Docker Compose 会读取当前目录下的 Dockerfile，根据它构建一个镜像，并用该镜像启动 app 服务容器。
  • ports: - 80:8080
    • 含义：将主机的 80 端口映射到容器的 8080 端口，格式为 主机端口:容器端口。
    • 作用：允许外部通过主机的 80 端口访问容器内运行在 8080 端口的应用（例如一个 Web 服务）。
  • volumes: - ./:/app
    • 含义：将主机的当前目录（./）挂载到容器的 /app 目录。
    • 作用：实现主机和容器之间的文件同步，适合开发环境，便于实时更新代码。
  • environment: - TZ=Asia/Shanghai
    • 含义：设置容器的环境变量 TZ（时区）为 Asia/Shanghai。
    • 作用：确保容器内的应用使用上海时区，避免时间显示错误。
b. redis:（Redis 服务）
  • image: redis:5.0.13
    • 含义：指定使用 Docker Hub 上的官方 Redis 镜像，版本为 5.0.13。
    • 作用：Docker Compose 会直接拉取这个镜像并启动 Redis 容器，无需额外的构建过程。
  • volumes: - redis:/data
    • 含义：将名为 redis 的卷挂载到容器的 /data 目录。
    • 作用：Redis 默认将数据存储在 /data 目录，通过卷挂载可以持久化数据，即使容器重启，数据也不会丢失。
  • environment: - TZ=Asia/Shanghai
    • 含义：设置 Redis 容器的时区为 Asia/Shanghai。
    • 作用：与 app 服务保持一致，确保时间记录统一。

3. volumes:

  • 含义：定义了 Docker Compose 管理的命名卷。
  • 作用：这里定义了一个名为 redis 的卷，供 redis 服务使用，用于持久化存储 Redis 的数据。
总结

Docker Compose 文件（通常命名为 docker-compose.yml）是用于定义和管理多容器 Docker 应用的配置文件。以下是编写它的基本结构和应该包含的内容：

基本结构

应该包含的内容

  1. version

    • 作用：指定 Docker Compose 文件的版本号，例如 “3.7”。
    • 注意：选择与你的 Docker 引擎版本兼容的版本，不同版本支持的功能不同。

  2. services

    • 作用：定义应用中的各个服务，每个服务对应一个容器。

    • 常用配置项

      ：

      • image：指定预构建的镜像（如 redis:5.0.13）。
      • build：指定构建镜像的路径（如 ./），需要搭配 Dockerfile。
      • ports：映射端口（如 80:8080），连接主机和容器。
      • volumes：挂载主机目录或卷（如 ./:/app 或 redis:/data），用于文件共享或数据持久化。
      • environment：设置环境变量（如 TZ=Asia/Shanghai），配置服务运行环境。
      • depends_on（可选）：指定服务依赖关系，控制启动顺序。
      • networks（可选）：指定服务连接的网络。

  3. volumes

    （可选）

    • 作用：定义命名卷（如 redis:），用于持久化数据或在容器间共享数据。
    • 类型：可以是命名卷（显式定义）或匿名卷（自动生成）。

  4. networks

    （可选）

    • 作用：定义自定义网络，控制服务之间的通信。
    • 配置：可以指定网络驱动（如 bridge）、子网等。

编写建议

  • 版本选择：根据你的 Docker 版本选择合适的 Compose 版本（如 “3.7”）。
  • 服务命名：使用简洁、有意义的名字，避免与关键字冲突。
  • 镜像或构建：需要自定义容器时用 build，否则用 image。
  • 端口管理：只映射必要的端口，避免冲突。
  • 卷使用：为需要持久化的服务配置卷（如数据库）。
  • 环境变量：通过 environment 配置灵活参数，避免硬编码。
  • 依赖关系：使用 depends_on 确保服务按顺序启动，但注意它不保证服务完全就绪。

在docker-compose.yml 文件所在目录，执行：docker-compose up就可以跑起来了。
命令参考：https://docs.docker.com/compose/reference/up/

在后台运行只需要加一个 -d 参数docker-compose up -d

其他命令

查看运行状态：docker-compose ps
停止运行：docker-compose stop
重启：docker-compose restart
重启单个服务：docker-compose restart service-name
进入容器命令行：docker-compose exec service-name sh
查看容器运行log：docker-compose logs [service-name]
